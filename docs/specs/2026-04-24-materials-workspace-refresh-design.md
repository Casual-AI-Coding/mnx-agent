# 素材管理与创作工作台刷新设计

> 日期: 2026-04-24
> 状态: Draft

## 背景

当前新增的素材管理界面存在两类严重问题：

1. 功能问题：歌曲详情中的“新建歌曲”“新建提示词”在特定场景下点击后没有可见反馈，用户感知为“没有弹窗，没有接口调用”。
2. 体验问题：`素材管理`、`新建素材`、`素材详情 / Artist Workspace` 三个界面的视觉语言过于同质，整体表现为发灰、发闷、层级弱、创作感不足。

本次设计目标不是做一次轻量 polish，而是在保留现有信息架构、共享组件与 token 体系的前提下，同时完成：

- 修复创建交互在空态下失效的问题
- 重建 materials 相关页面的视觉层次与创作气质
- 为后续实现补齐可验证的组件测试与本地无头浏览器验证路径

---

## 1. 已确认根因

### 1.1 “新建歌曲”无响应的根因

文件：`src/components/materials/artist/SongLibraryPanel.tsx`

- 当 `songs.length === 0` 时，组件在空态分支提前 `return`
- 空态按钮点击只执行 `setIsCreating(true)`
- 真正的 `Dialog open={isCreating}` 仅存在于非空态返回体底部
- 因此空态下虽然状态被设为 `true`，但弹窗本身没有进入渲染树

结论：这是组件分支结构导致的渲染缺失，不是点击事件未绑定，也不是接口未调用。

### 1.2 “新建提示词”无响应的根因

文件：`src/components/materials/artist/SongPromptPanel.tsx`

- 当 `prompts.length === 0` 时，组件在空态分支提前 `return`
- 空态按钮点击只执行 `setIsCreating(true)`
- 真正的 `Dialog open={isCreating}` 仅存在于非空态返回体底部
- 因此当前歌曲没有 prompt 时，点击“新建提示词”不会出现任何可见弹窗

结论：根因与 `SongLibraryPanel` 同构，属于空态分支提前返回造成的共享创建弹窗未渲染。

### 1.3 为什么不是 Dialog 组件本身的问题

已有代码与测试说明共享 `Dialog` 本身可工作：

- `src/components/ui/Dialog.tsx` 使用 portal 将内容挂到 `document.body`
- `src/components/materials/__tests__/MaterialManagementLayout.test.tsx` 已验证“创建素材”对话框会渲染到 `document.body`

因此，本次功能问题的修复重点应放在 **panel 组件的渲染结构**，而不是重写 Dialog 基础设施。

## 2. 设计目标

### 2.1 功能目标

- `新建素材`、`新建歌曲`、`新建提示词` 在空态与非空态下都能稳定打开弹窗
- 点击后必须有即时、可见、可理解的反馈
- 提交中、提交成功、提交失败都要有明确反馈，不再出现“像没反应”

### 2.2 视觉目标

- 把 materials 相关页面从“普通后台卡片页”提升为“编辑部式暗色创作工作台”
- 保留现有 dark token 基底，不推翻现有组件体系
- 让每个主 panel 有明确身份，而不是一组同质的灰黑盒子
- 让创建动作真正成为创作入口，而不是角落里的普通按钮

### 2.3 工程目标

- 保留现有路由与 2x2 工作台结构
- 不引入新的重型 UI 框架
- 通过新增测试覆盖本次 bug 的真实触发场景
- 通过本地无头浏览器完成运行时核验，不依赖用户访问本地 URL

---

## 3. 整体视觉方向

### 3.1 视觉定位

整体定位为：**编辑部式的暗色创作工作台**。

这不是通用 SaaS 黑卡片，也不是高饱和赛博风，而是：

- 深色底，保留当前系统稳定的暗色基底
- 明确分区，让每个工作区域一眼可辨
- 强标题与节奏感，让页面更像在整理和创作内容，而不是填表
- 少量高冲击强调色，用来承接“创建”“当前选中”“默认项”等关键状态

### 3.2 记忆点

本次改造的核心记忆点是：**四个主 panel 都有自己的身份感**。

用户应该能够在进入页面的第一眼就区分：

- 这是档案区
- 这是人物创作提示区
- 这是歌曲编目区
- 这是当前歌曲的风格工作区

而不是只看到四个长得差不多的深色卡片。

### 3.3 不采用的方向

本次明确不采用以下方向：

- 不做纯浅色改造
- 不做大面积赛博霓虹
- 不做纯玻璃拟态堆叠
- 不脱离当前 token 体系另起一套视觉系统

---

## 4. 页面级设计

## 4.1 素材管理页：素材策展台

页面定位从“列表页”提升为“素材策展台”。

### 结构层次

页面保持现有功能结构，但视觉上拆成三层：

1. 页面级入口区：标题、说明、主创建动作
2. 工具区：搜索、数量、状态提示
3. 内容区：素材列表 / 空态

### 视觉策略

- 顶部主创建按钮作为明确的主行动入口，不再只是普通 header button
- 搜索与数量区从单一灰卡片提升为更清晰的工具带
- 列表项强化“封面感”，让素材名称、类型、更新时间形成清晰主次
- 类型标记不再只是陪衬徽章，而是帮助用户快速理解内容类型的识别点

### 列表项改造重点

- 增强名称、描述、更新时间之间的层级关系
- 增强 hover 与聚焦状态，让“继续编辑”意图更明确
- 让操作区在默认状态下更克制、hover 时更明确，避免整行显得挤
- 保留现有功能：搜索、创建、删除、进入编辑页

### 空态策略

空态不再只是“暂无素材 + 一个按钮”，而要表现为“开始建立创作资产”的入口。

---

## 4.2 新建素材弹窗：起稿面板

新建素材弹窗从“通用表单弹窗”改为“起稿面板”。

### 目标

- 让用户理解这是创建一个新的创作容器，而不是提交一条普通后台记录
- 在不增加字段复杂度的前提下，提高弹窗的开始感与动作确定性

### 视觉与文案策略

- 标题区更强，说明文案更短、更准
- `素材类型` 作为第一步决策，视觉上更像起始选择，而不是普通 select 字段
- `素材名称` 保持主要输入焦点
- CTA 文案保持“创建素材”，但其位置、重量和前置说明都应更明确

### 保留边界

- 当前只保留 `artist` 类型，不做多类型扩展 UI
- 不增加额外步骤页
- 不引入 wizard 流程

---

## 4.3 素材详情页：Artist Workspace

`ArtistWorkspace` 保留当前已验证合理的 2x2 结构，不推翻信息架构；本次重点是让四个 panel 的身份、节奏和创建动作显著区分。

### 4.3.1 基本信息面板

定位为：**档案卡**。

- 视觉上更像主体档案，而不是普通表单区
- 强化标题、说明与字段的主次
- 保存动作应体现“更新人物档案”而不是模糊的局部提交

### 4.3.2 音乐人风格 Prompt 面板

定位为：**创作母板**。

- 这是人物级的风格定义，不应与歌曲层的 prompt 区混淆
- 保持 tab/candidate 结构，但增强默认项、候选关系和编辑焦点
- 文案与视觉都应强调“这是人物全局风格参考”

### 4.3.3 歌曲库面板

定位为：**编目区**。

- 强化歌曲列表的编排感和被选中状态
- “新建歌曲”应像加入创作条目，而不是简单添加一行记录
- 空态要明确告诉用户：先创建歌曲，后续才能写歌曲级 prompt

### 4.3.4 歌曲风格 Prompt 面板

定位为：**当前歌曲的风格工作区**。

- 必须清楚表达“这个区域服务于当前选中的歌曲”
- 与音乐人级 prompt 做出明确视觉区分
- 空态必须让用户理解：当前歌曲还没有风格候选，现在可以创建第一条

---

## 5. 创建交互统一设计

## 5.1 统一原则

`新建素材`、`新建歌曲`、`新建提示词` 无论处于空态还是非空态，都要满足三个统一原则：

1. 点击后立即出现可见反馈
2. 弹窗结构、标题区、主按钮语义保持一致
3. 创建成功后，内容落点清楚，用户知道新内容出现在哪里

### 5.2 空态与非空态的差别

- 空态：强调“进入创作”
- 非空态：强调“继续扩展”

两者可以在说明文案和按钮位置上有差别，但不能出现一个是正式入口、一个像临时补丁的感觉。

### 5.3 结构改动结论

`SongLibraryPanel` 与 `SongPromptPanel` 的创建弹窗都要被提升到各自组件的共享渲染层。

也就是说：

- 空态分支可以继续单独返回视觉内容
- 但 `Dialog` 不能只写在非空态返回体内部
- 需要改成“空态和非空态都能共享到同一套创建弹窗渲染”

这是本次功能修复的核心结构要求。

---

## 6. 反馈层次设计

为彻底消除“点了像没反应”的感受，创建链路必须具备完整反馈层次。

### 6.1 点击反馈

- 点击后立刻出现弹窗
- 弹窗标题和说明要足够明确，用户知道自己进入了哪种创建动作

### 6.2 提交中反馈

- 主按钮进入 loading 态
- 相关提交按钮禁用
- 避免重复点击与重复创建

### 6.3 成功反馈

- toast 保留
- 列表或 tab 内容要即时刷新
- 新建歌曲后应有明确选中落点
- 新建提示词后应能立即在当前候选中看到新增内容

### 6.4 失败反馈

- 保留已有错误提示文案机制
- 失败时保持弹窗上下文，便于用户修正后重试

## 7. 测试与验证设计

## 7.1 组件测试

需要在现有测试基础上补充本次 bug 的最小回归保护。

### 必补用例

文件：`src/components/materials/artist/__tests__/ArtistWorkspace.test.tsx`

至少新增以下覆盖：

1. 当歌曲列表为空时，点击“新建歌曲”后应出现标题为“新建歌曲”的 dialog
2. 当当前歌曲存在但 prompt 列表为空时，点击“新建提示词”后应出现标题为“新建提示词”的 dialog

### 目的

- 防止未来再次因为空态提前 `return` 造成弹窗不渲染

除了组件测试，还需要确认 `ArtistWorkspace` 在真实组合状态下没有新的结构性回归：

- 新建歌曲后歌曲面板与歌曲 prompt 面板联动正常
- 当前歌曲无 prompt 时，歌曲 prompt 面板仍能打开创建弹窗

## 7.3 无头浏览器验证

按照用户约束，本次运行时验证必须由我本地完成，不要求用户访问任何本地 URL。

### 验证范围

- `素材详情 / Artist Workspace`
- `新建歌曲` 创建链路

### 验证目标

## 8. 实现边界

### 必须保持

- 保留现有路由：`/materials`、`/materials/:id/edit`
- 保留 `ArtistWorkspace` 的 2x2 信息结构
- 保留共享 `Dialog`、`Card`、`Button` 组件体系
- 保留当前 Tailwind 语义 token 体系

### 本次不做

- 不重构为多步向导
- 不新增素材类型体系 UI
- 不引入新的大型设计系统依赖
- 不改动后端数据模型
- 不扩展到模板系统或工作流接入

---

## 9. 验收标准

本次设计完成后，应满足以下标准：

1. 空态下点击“新建歌曲”必定出现创建弹窗
2. 空态下点击“新建提示词”必定出现创建弹窗
3. materials 相关三个界面不再呈现高度同质的灰黑卡片堆叠感
4. 四个主 panel 具有清晰身份与视觉分区
5. 创建动作在空态与非空态下保持统一的入口语言与行为反馈
6. 至少新增两条针对空态创建弹窗的回归测试
7. 由本地无头浏览器完成运行时验证，不依赖用户访问本地地址

---

## 10. 实施建议顺序

推荐按以下顺序实施：

1. 先修复 `SongLibraryPanel` / `SongPromptPanel` 的共享弹窗渲染结构
2. 立即补充对应组件测试，先把 bug 钉住
3. 再进入 `素材管理页`、`新建素材弹窗`、`Artist Workspace` 的视觉改造
4. 最后使用本地无头浏览器做实际页面核验

这个顺序可以先消除功能性断点，再推进视觉升级，降低回归风险。
