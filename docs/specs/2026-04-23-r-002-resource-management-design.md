# R-002 资源管理重定义设计

## 1. 背景与目标

### 1.1 背景
- 现有 `media_records` / `MediaManagement` 管理的是生成或上传后的媒体文件，不适合承载手工录入的结构化文本素材。
- 现有模板体系分为 `prompt_templates` 与 `workflow_templates` 两套，但本次讨论的范围属于**资源管理域**，不是工作流模板管理。
- 本期只做管理能力，不做调试/工作流中的使用流程集成。

### 1.2 R-002 重定义结论
R-002 拆分为两期：

- **Phase 1：素材管理**
- **Phase 2：模板管理重做**

本文仅覆盖 **Phase 1：素材管理**。

### 1.3 Phase 1 目标
交付一个面向未来扩展的素材管理底座，满足以下目标：

- 后端采用**通用素材主表 + 通用素材子表**的结构化模型
- 前端采用**统一资源管理容器 + 类型专属编辑器**的交互模式
- 首个、且唯一落地的顶层素材类型为 **音乐人**
- 支持素材主实体、子实体，以及**独立的跨场景通用 Prompt 能力**

---

## 2. 范围定义

### 2.1 本期包含
- 新增素材管理能力
- 素材主表与素材子表的数据结构及管理接口
- 首个素材类型 `音乐人`
- `音乐人` 下子实体管理
- 独立通用 Prompt 表设计与管理能力
- 音乐人级 Prompt 候选管理
- 歌曲级 Prompt 候选管理
- 基础搜索、筛选、排序、默认项管理
- 面向未来扩展其他素材类型的底座设计

### 2.2 本期不包含
- 模板管理实现
- 调试/工作流中的素材接入与运行时使用流程
- 工作流模板管理重构
- 批量导入导出
- 复制/批量操作
- 发布流、审核流、回滚
- Prompt 版本树、Diff 对比、历史回滚
- 全局 Prompt 管理中心页面

### 2.3 Phase 2 前置约束
- 模板允许引用素材
- Prompt 是跨场景通用能力，不绑定素材域专表
- Phase 1 的素材模型、子素材模型、Prompt 归属模型，必须能被未来模板系统稳定引用

---

## 3. 核心方案结论

### 3.1 方案结论
采用：

**通用素材主表 + 通用素材子表 + 独立通用 Prompt 表 + 音乐人专属编辑器**

### 3.2 采用原因
- 符合“后端通用底座 + 前端按类型定制”的目标
- 首期只落地 `音乐人`，但底层模型不被写死成音乐专用表
- 比单表 JSON 更适合后续做子项管理、默认项管理、模板引用
- 比音乐人专属表群更利于未来扩展其他素材类型
- 将 Prompt 提升为独立通用能力后，后续其他业务对象也可复用同一套归属模型

---

## 4. 领域模型

### 4.1 一期领域模型
一期采用三层模型：

1. **素材主实体**
2. **素材子实体**
3. **通用 Prompt 实体**

### 4.2 首个素材类型：音乐人
本期首个、且唯一落地的顶层素材类型为：

- `音乐人`

以下内容都**不是并列顶层素材类型**，而是 `音乐人` 内部的结构化内容：

- 音乐人风格 Prompt
- 歌曲
- 歌曲风格 Prompt
- 歌词

### 4.3 音乐人素材语义
一个音乐人素材是围绕“音乐人”组织的创作参考单元，包含：

- 音乐人级基础信息
- 音乐人级风格 Prompt 候选
- 歌曲库
- 每首歌曲的歌词
- 每首歌曲的风格 Prompt 候选

### 4.4 子实体建模结论
素材子实体必须具备类型字段。

虽然本期 `音乐人` 下只启用一种子类型：

- `歌曲`

但模型上必须为未来其他子类型保留扩展位。

### 4.5 Prompt 建模结论
Prompt 不是素材域的附属表，而是**独立的跨场景通用能力表**。

归属通过以下维度表达：

- `target_type`
- `target_id`
- `slot_type`

本期“版本 / 变体”不区分语义，统一建模为：

- **同一槽位下的多个候选 Prompt**

不设计版本树。

## 5. 数据结构设计

### 5.1 总体结构
本期采用三张核心表：

1. 素材主表
2. 素材子表
3. 通用 Prompt 表

### 5.2 素材主表职责
每条主表记录表示一个素材主体。

本期对应关系：

- 一条主表记录 = 一个音乐人素材

主表承载：

- `id`
- `material_type`
- `name`
- `description`
- `owner_id`
- `sort_order`
- `created_at` / `updated_at`
- `is_deleted` / `deleted_at`
- 主体级扩展信息

本期主体级内容重点包括：

- 音乐人名称
- 音乐人描述
- 音乐人补充信息

### 5.3 素材子表职责
每条子表记录表示某个主素材下的结构化子项。

本期对应关系：

- `音乐人` 下的子实体 = `歌曲`

子表承载：

- `id`
- `material_id`
- `item_type`
- `name`
- `lyrics`
- `sort_order`
- `remark` / 扩展信息
- `owner_id`
- `created_at` / `updated_at`
- `is_deleted` / `deleted_at`

### 5.4 通用 Prompt 表职责
通用 Prompt 表承载：

- 某个目标对象的某个 Prompt 槽位
- 该槽位下的多个候选 Prompt

它既可以挂在：

- 素材主实体
- 素材子实体

未来也可挂在：

- 模板
- 其他业务对象

### 5.5 通用 Prompt 表公共字段
建议最小公共字段为：

- `id`
- `target_type`
- `target_id`
- `slot_type`
- `name`
- `content`
- `sort_order`
- `is_default`
- `owner_id`
- `created_at` / `updated_at`
- `is_deleted` / `deleted_at`

### 5.6 关系定义
- 素材主表 `1 : N` 素材子表
- 素材主表 `1 : N` 通用 Prompt 表
- 素材子表 `1 : N` 通用 Prompt 表

本期在 `音乐人` 类型下的映射：

- 一个音乐人 → 多首歌曲
- 一个音乐人 → 多条音乐人风格 Prompt 候选
- 一首歌曲 → 多条歌曲风格 Prompt 候选

### 5.7 为什么不采用单表 JSON 内嵌
- 子项单独更新困难
- 默认项与排序管理脆弱
- 后续模板引用边界不清晰
- 检索、唯一性约束、归属校验难以稳定落地

结论：

- 主体、子项、Prompt 候选都应结构化存储

## 6. API 设计

### 6.1 素材主实体 API
负责：

- 素材列表查询
- 详情查询
- 创建
- 更新
- 软删除

列表至少支持：

- 按素材类型过滤
- 按关键词搜索
- 分页
- 排序
- 状态过滤（若实现）

虽然本期只有 `音乐人`，接口仍需保留类型参数，不将 API 写死成音乐人专用接口。

### 6.2 素材子实体 API
负责：

- 查询某个主素材下的子项列表
- 新增子项
- 更新子项
- 删除子项
- 排序

子项是从属某个主素材的对象，不设计为全局游离资源。

### 6.3 通用 Prompt API
负责：

- 查询某目标对象、某槽位下的候选 Prompt 列表
- 新增候选
- 更新候选
- 删除候选
- 设置默认项
- 调整排序

归属维度：

- `target_type`
- `target_id`
- `slot_type`

示例：

- `target_type = material-main` + `slot_type = artist-style`
- `target_type = material-item` + `slot_type = song-style`

### 6.4 聚合详情接口
需要提供一个聚合详情读接口，供前端专属编辑器一次性取回完整编辑数据。

对于一个音乐人详情，应一次返回：

- 主体信息
- 音乐人风格 Prompt 候选
- 歌曲列表
- 每首歌曲的歌曲风格 Prompt 候选

原则：

- 写接口保持职责清晰
- 读接口允许聚合，以支撑专属编辑体验

### 6.5 API 通用原则
- 路由使用复数名词
- 所有处理器使用 `asyncHandler`
- Zod 做第一层校验
- 响应格式统一为 `{ success, data/error }`
- 数据隔离依赖 `owner_id`
- SQL 必须参数化
- 删除统一采用软删除

## 7. 前端页面与交互设计

### 7.1 总体模式
采用：

- **统一资源管理容器 + 类型专属编辑器**

统一容器负责：

- 列表
- 搜索
- 筛选
- 入口管理

`音乐人` 类型使用专属编辑器承载复杂结构化编辑。

### 7.2 页面层级
建议两层核心界面：

1. 素材管理列表页
2. 音乐人专属编辑器

### 7.3 列表页职责
负责：

- 素材列表展示
- 搜索
- 筛选
- 新建入口
- 编辑入口
- 删除入口

建议展示字段：

- 素材名称
- 素材类型
- 子项数量（如歌曲数）
- 更新时间

### 7.4 音乐人专属编辑器布局
采用 **2×2 工作台布局**：

#### 左上：基本信息
- 音乐人名称
- 描述 / 备注
- 其他主体字段

#### 左下：音乐人风格 Prompt
- 候选 Prompt 列表
- 使用 tab 切换候选
- 支持新增、删除、编辑、设默认

#### 右上：歌曲库
- 歌曲列表
- 选中当前歌曲
- 支持新增、编辑、删除、排序

#### 右下：歌曲风格 Prompt
- 跟随右上当前选中歌曲变化
- 使用 tab 切换候选
- 支持新增、删除、编辑、设默认

### 7.5 歌曲库展示结论
歌曲库区不适合简单表格，更适合作为可管理的结构化列表。

原因是每首歌曲不仅有标题，还有歌词和其对应的 Prompt 候选。

### 7.6 Prompt 区交互结论
无论是音乐人级还是歌曲级，Prompt 区交互统一为：

- tab 切换候选
- 编辑当前候选正文
- 新增候选
- 删除候选
- 设置默认候选
- 调整顺序

### 7.7 本期前端明确不做
- schema 驱动万能动态表单
- 批量导入
- Diff 对比
- 历史版本查看
- 运行时引用预览
- 全局 Prompt 管理中心

## 8. 校验、约束与删除策略

### 8.1 主实体校验
主实体至少要求：

- 名称必填
- 类型必填
- `owner_id` 必填
- 软删除

同一 owner 下主实体名称本期不做强唯一约束。

### 8.2 子实体校验
子实体至少要求：

- 必须绑定主实体
- 子实体类型必填
- 名称必填
- 排序字段存在
- 软删除

关键约束：

> 同一主素材下，同类型子实体名称不可重名。

一期具体化为：

> 同一个音乐人下，歌曲名必须唯一。

该约束需同时体现在：

- 数据库约束
- 后端校验
- 前端错误提示

### 8.3 Prompt 校验
每条 Prompt 至少要求：

- `target_type` 必填
- `target_id` 必填
- `slot_type` 必填
- `content` 必填
- `sort_order` 必填
- `is_default` 必填

槽位级约束：

> 同一 `target + slot_type` 下，只允许一个默认项。

### 8.4 默认项删除策略
删除默认项时：

- 若该槽位仍有其他候选，则自动将排序最前的一条设为默认
- 若没有其他候选，则该槽位进入“无默认项”状态

### 8.5 删除策略
统一软删除：

- 主实体软删除
- 子实体软删除
- Prompt 记录软删除

联动策略：

#### 删除音乐人
- 音乐人软删除
- 其下歌曲子项软删除
- 关联到音乐人和其歌曲的 Prompt 一并软删除

#### 删除歌曲
- 歌曲软删除
- 该歌曲关联 Prompt 一并软删除

#### 删除 Prompt
- 仅删除该 Prompt 自身

### 8.6 排序规则
需要持久化排序的对象：

- 主实体列表
- 同一主实体下的子项
- 同一槽位下的 Prompt 候选

排序字段必须由后端持久化，不依赖前端临时顺序。

### 8.7 搜索与筛选范围
本期基础搜索筛选仅覆盖主列表：

- 按名称搜索
- 按素材类型过滤
- 按状态过滤（若实现）
- 按更新时间排序

本期不做：

- 跨歌词全文搜索
- 跨 Prompt 全文搜索
- 高级组合检索

### 8.8 并发保护
本期不做复杂协同锁，但建议保留基础并发保护能力：

- 基于 `updated_at` 或版本号做基础乐观保护

## 9. 测试策略与验收标准

### 9.1 后端测试重点

#### 素材主实体
- 创建 `音乐人` 类型素材成功
- 更新成功
- 列表分页、搜索有效
- 软删除后默认不可见
- `owner_id` 隔离有效

#### 素材子实体
- 可在某音乐人下创建歌曲子项
- 可更新歌曲内容
- 可排序
- 可删除
- 同一音乐人下歌曲重名时报错
- 不同音乐人下允许同名歌曲

#### 通用 Prompt
- 可挂在主实体
- 可挂在子实体
- 同槽位可有多个候选
- 同槽位只能有一个默认项
- 删除默认项时自动迁移或清空默认项
- 排序生效

#### 聚合详情接口
- 返回主体信息
- 返回主体级 Prompt
- 返回子项列表
- 返回子项级 Prompt
- 空数据场景可正常返回
- 软删除数据不会混入

### 9.2 前端测试重点

#### 列表页
- 列表加载成功
- 搜索 / 筛选生效
- 可进入创建 / 编辑
- 删除后刷新正确

#### 音乐人专属编辑器
- 左上基本信息可编辑保存
- 左下音乐人风格 Prompt tab 切换正确
- 右上歌曲库可新增 / 编辑 / 删除 / 排序
- 切换当前歌曲后，右下 Prompt 正确联动
- 右下 Prompt tab 切换和默认项切换正确

#### 表单约束
- 重名歌曲被正确拦截并提示
- 必填项缺失时有明确提示
- 删除默认 Prompt 后 UI 状态正确更新

### 9.3 功能验收标准
- 能新增、编辑、删除音乐人素材
- 能在音乐人下管理歌曲子项
- 能管理音乐人级 / 歌曲级 Prompt 候选
- 能设置默认 Prompt
- 能对歌曲与 Prompt 候选排序
- 同一音乐人下禁止重名歌曲

### 9.4 数据验收标准
- 主表 / 子表 / 通用 Prompt 表关系清晰
- 全部数据带 `owner_id`
- 全部删除为软删除
- Prompt 通过 `target_type + target_id + slot_type` 归属，不绑定单一业务专表

### 9.5 UI 验收标准
- 编辑器采用确认后的 2×2 工作台布局
- 左下、右下 Prompt 区使用 tab 切换候选
- 右上切换歌曲时，右下内容正确切换
- 错误提示明确，无静默失败

### 9.6 工程验收标准
- 前端测试通过
- 后端测试通过
- `npm run build` 通过
- `npm run test:coverage` 满足现有阈值

### 9.7 本期成功标准
系统能够以通用素材框架承载首个 `音乐人` 类型素材，并提供稳定的主实体、子实体、独立通用 Prompt 管理能力，以及符合 2×2 工作台布局的专属编辑体验。

- 通用素材主表、子表、Prompt 表的准确命名
